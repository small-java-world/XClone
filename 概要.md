## 2. システム概要

本システムは、X（旧Twitter）のクローンアプリを実装するための仕様書です。  
バックエンドは Kotlin 2.x + Spring Boot を用い、jOOQ による型安全なデータアクセスと kotlin‑logging による構造化ログ出力（JSON形式）を実現します。  
各バウンデッドコンテキスト（ユーザ、ツイート、通知）では、DDD の原則に従い、以下の各層に分離して実装します。

- **Adapter 層**  
  - Controller（REST API）およびイベントリスナー（@EventListener）を配置  
  - イベントリスナーは発行されたドメインイベントを受信し、統合用ドメインサービスへ委譲する
- **Domain 層**  
  - エンティティ、値オブジェクト、ドメインイベント、専用ファクトリー（UserFactory、TweetFactory、NotificationFactory）を配置  
  - リポジトリのインターフェースもこの層に配置する（例：`com.example.app.user.repository.UserRepository.kt`）
- **Usecase 層**  
  - 各種コマンドオブジェクト（RegisterUserCommand、PostTweetCommand、CreateNotificationCommand など）を受け、ドメイン層の処理を呼び出す単一責務クラス
- **Repository／Infrastructure 層**  
  - 永続化の実装は、各バウンデッドコンテキストのインフラストラクチャ層に配置する  
  - 例：ユーザリポジトリ実装は `com.example.app.user.infrastructure.repository.UserRepositoryImpl.kt`
- **Integration 層**  
  - 複数バウンデッドコンテキスト間の連携を実現するため、統合用ドメインサービス（例：TweetNotificationIntegrationService）を介して実装

フロントエンドは Next.js (React) を採用し、Container/Presentational パターンで各画面（ホーム、ユーザ、ツイート、通知）を整理。  
通知はプッシュ方式ではなく、定期的なポーリング（useNotificationPolling フック）により REST API を呼び出して取得し、ユーザ操作（タップ）で既読／削除処理を実施します。

バックエンドの単体テストは Kotest と Mockk、フロントエンドの UT は Vitest、E2E テストは Playwright を用いてそれぞれ実施します。

## 3. ドメインモデルと主要要素

### 3.1 バウンデッドコンテキスト

#### ユーザコンテキスト
- **集約**: User (Aggregate Root)

  - UserRegisteredEvent  
  - UserProfileUpdatedEvent

#### ツイートコンテキスト
- **集約**: Tweet (Aggregate Root)  
  - **内部要素**:  
    - Like, Retweet, Media（各エンティティ）
- **値オブジェクト**: TweetText, MediaUrl, MediaType  
- **ドメインイベント**:  
  - TweetPostedEvent  
  - TweetEditedEvent  
  - TweetDeletedEvent

#### 通知コンテキスト
- **集約**: Notification (Aggregate Root)
- **値オブジェクト**: NotificationType, NotificationContent  
- **ドメインイベント**:  
  - NotificationCreatedEvent  
- **特記事項**:  
  - 通知はプッシュ方式ではなく、API 経由で返され、クライアントは定期的なポーリングで取得

### 3.2 エンティティ生成とファクトリパターン
- 各エンティティは以下のファクトリメソッドを内部に実装：
  - **create**: XXXCommand を受け、値オブジェクトによるバリデーション済みの上で新規インスタンスを生成
  - **reconstructor**: 永続化レコードからエンティティを再構築
  - **copy**: イミュータブルな更新のための新たなインスタンスを生成
- 複雑な生成ロジックが必要な場合は、専用ファクトリークラス（UserFactory、TweetFactory、NotificationFactory）で実装

## 4. ドメインイベントと統合用ドメインサービス

### 4.1 ドメインイベントの発行
- 各集約は、状態変化に応じたドメインイベント（例：TweetPostedEvent、UserRegisteredEvent、NotificationCreatedEvent）を発行

### 4.2 統合用ドメインサービス
- **目的**:  
  - TweetPostedEvent の情報をもとに、通知生成処理を統括する統合用ドメインサービス（TweetNotificationIntegrationService）を導入する  
  - このサービスが、ツイート投稿イベントと通知生成処理の橋渡しを行い、各コンテキスト間の直接依存を回避する
- **配置**:  
  - `com.example.app.integration` パッケージに配置
- **実装例**:

  ```kotlin
  // backend/src/main/kotlin/com/example/app/integration/TweetNotificationIntegrationService.kt
  package com.example.app.integration

  import com.example.app.tweet.domain.event.TweetPostedEvent
  import com.example.app.notification.usecase.NotificationUsecase
  import com.example.app.notification.usecase.CreateNotificationCommand
  import mu.KotlinLogging
  import org.springframework.stereotype.Service

  private val logger = KotlinLogging.logger {}

  @Service
  class TweetNotificationIntegrationService(
      private val notificationUsecase: NotificationUsecase
  ) {
      fun processTweetPosted(event: TweetPostedEvent) {
          logger.info {
              """{
                "timestamp": "${System.currentTimeMillis()}",
                "level": "INFO",
                "eventType": "TweetPostedEvent",
                "operation": "TweetNotificationIntegrationService.processTweetPosted",
                "userId": "${event.postedBy.take(4)}****",
                "message": "統合サービスでTweetPostedEventを受信"
              }""".trimIndent()
          }
          val command = CreateNotificationCommand(
              userId = event.postedBy,
              content = "新しいツイートが投稿されました！"
          )
          notificationUsecase.createNotification(command)
          logger.info { "統合サービスでのTweetPostedEvent処理完了" }
      }
  }
  ```

### 4.3 イベントリスナーの設計と配置
- **配置**:  
  - イベントリスナーは Adapter 層に配置し、統合用ドメインサービスを呼び出すことで直接依存を回避する
- **実装例（TweetEventListener）**:

  ```kotlin
  // backend/src/main/kotlin/com/example/app/tweet/adapter/event/TweetEventListener.kt
  package com.example.app.tweet.adapter.event

  import com.example.app.tweet.domain.event.TweetPostedEvent
  import com.example.app.integration.TweetNotificationIntegrationService
  import mu.KotlinLogging
  import org.springframework.context.event.EventListener
  import org.springframework.stereotype.Component

  private val logger = KotlinLogging.logger {}

  @Component
  class TweetEventListener(
      private val integrationService: TweetNotificationIntegrationService
  ) {
      @EventListener
      fun onTweetPosted(event: TweetPostedEvent) {
          logger.info { "TweetEventListener: TweetPostedEventを受信" }
          integrationService.processTweetPosted(event)
      }
  }
  ```

- **その他のイベントリスナー**（UserEventListener、NotificationEventListener）は各コンテキスト内で実装（前述のサンプルコード参照）

## 5. 通知ポーリングの仕組み（フロントエンド側）

### 5.1 ポーリングフック (useNotificationPolling)
- 定期的に API を呼び出して最新の通知情報を取得するカスタムフック

  ```typescript
  // frontend/hooks/useNotificationPolling.ts
  import { useEffect, useState } from 'react';
  import apiClient from '../utils/apiClient';

  export interface Notification {
    id: string;
    type: string;
    content: string;
    // その他必要な属性
  }

  export function useNotificationPolling(intervalMs: number = 30000): Notification[] {
    const [notifications, setNotifications] = useState<Notification[]>([]);

    useEffect(() => {
      const fetchNotifications = () => {
        apiClient.get<Notification[]>('/notifications')
          .then((data) => setNotifications(data))
          .catch((err) => console.error('Failed to fetch notifications:', err));
      };

      fetchNotifications();
      const timer = setInterval(fetchNotifications, intervalMs);
      return () => clearInterval(timer);
    }, [intervalMs]);

    return notifications;
  }
  ```

### 5.2 NotificationContainer
- Container コンポーネントがポーリングフックを利用して通知一覧の状態管理および、タップ時の既読／削除 API 呼び出しを実施

  ```tsx
  // frontend/components/NotificationScreen/containers/NotificationContainer.tsx
  import React, { useState, useEffect } from 'react';
  import { useNotificationPolling, Notification } from '../../../hooks/useNotificationPolling';
  import NotificationList from '../presentational/NotificationList';
  import apiClient from '../../../utils/apiClient';

  const NotificationContainer: React.FC = () => {
    const polledNotifications = useNotificationPolling();
    const [notifications, setNotifications] = useState<Notification[]>([]);

    useEffect(() => {
      setNotifications(polledNotifications);
    }, [polledNotifications]);

    const handleNotificationTap = (id: string) => {
      apiClient.post(`/notifications/${id}/read`)
        .then(() => setNotifications((prev) => prev.filter((n) => n.id !== id)))
        .catch((err) => console.error('Failed to mark notification as read:', err));
    };

    return (
      <NotificationList notifications={notifications} onNotificationTap={handleNotificationTap} />
    );
  };

  export default NotificationContainer;
  ```

### 5.3 NotificationList (Presentational)
- 単純に通知一覧を表示し、タップ時にハンドラーを呼び出す UI コンポーネント

  ```tsx
  // frontend/components/NotificationScreen/presentational/NotificationList.tsx
  import React from 'react';
  import { Notification } from '../../../hooks/useNotificationPolling';

  interface Props {
    notifications: Notification[];
    onNotificationTap: (id: string) => void;
  }

  const NotificationList: React.FC<Props> = ({ notifications, onNotificationTap }) => (
    <ul>
      {notifications.map((notification) => (
        <li key={notification.id} onClick={() => onNotificationTap(notification.id)}>
          <strong>{notification.type}</strong>: {notification.content}
        </li>
      ))}
    </ul>
  );

  export default NotificationList;
  ```

## 6. ログ出力とデバッグ運用（kotlin‑logging 採用）

### 6.1 ログ出力ポリシー
- **ログレベル**:  
  - **Infoログ**: ユースケースの開始／完了、イベント発行など業務処理の進捗を記録。ただし、個人情報（メールアドレス、パスワード、詳細プロフィール情報）はマスクまたは除外する  
  - **Error/Warningログ**: エラー発生時の例外情報、スタックトレース等を記録（PII は含まない）
- **構造化ログ**:  
  - 全ログを JSON 形式などの構造化形式で出力し、各ログエントリに eventType、operation、マスク済みユーザID、timestamp などの情報を含む  
  - 例:
    ```json
    {
      "timestamp": "2024-09-15T12:34:56Z",
      "level": "INFO",
      "eventType": "TweetPostedEvent",
      "operation": "TweetUsecase.createTweet",
      "userId": "user-XXXX",
      "message": "新規ツイート投稿処理開始"
    }
    ```
- **AIエージェント連携**:  
  - 生成AIエージェントがログを解析し、異常パターンやエラー箇所を自動検知できるよう、詳細な情報を出力する

## 7. テスト戦略

### 7.1 バックエンドの単体テスト（UT）
- **テストツール**: Kotest、Mockk  
- **対象**:  
  - 各層（Adapter、Domain、Usecase、Repository／Infrastructure）の単体テスト  
  - **値オブジェクトのテスト**:  
    - 各バウンデッドコンテキストの `src/test/kotlin/com/example/app/{context}/domain/valueobject/` 配下に、  
      - EmailAddressTest.kt  
      - UserNameTest.kt  
      - PasswordTest.kt  
      - UserProfileTest.kt  
      などを配置し、生成時のバリデーション、不変性、マスク処理を検証

### 7.2 フロントエンドの単体テスト（UT）
- **テストツール**: Vitest  
- **対象**:  
  - 各画面ごとに、Container コンポーネントおよび Presentational コンポーネントの単体テスト  
  - テストコードは `frontend/tests` 以下に、各画面（HomeScreen, UserScreen, TweetScreen, NotificationScreen）ごとに分離

### 7.3 E2E テスト
- **テストツール**: Playwright  
- **対象**:  
  - バックエンドとフロントエンドの連携を含むユーザシナリオ（例：ユーザ登録、ツイート投稿、通知取得、通知タップによる既読処理）を検証  
  - E2E テストは、`e2e/` ディレクトリに独立したテストプロジェクトとして管理

### 7.4 テストデータ投入ガイドライン（バックエンド）
- **テスト用DB利用**: 単体テスト実行時は専用のテスト用DBを使用し、実際のSQLを発行して検証する。
- **共通テストデータの準備**:  
  - 事前に定義された共通テストデータのSQLファイル（例：`src/test/resources/common_test_data.sql`）を用意する。
- **テスト前処理手順**:
  1. JDBC経由でDBの参照整合性を一時無効化する。
  2. 既存のテストデータを全削除するSQLを実行する。
  3. 共通テストデータSQLファイルを実行し、テスト用DBにデータを投入する。
  4. 各テストケースで必要な追加データは、テストケース内で投入する。
  5. 全データ投入後、DBの参照整合性を再有効化する。
- **投入方法**:  
  - テストデータの投入は、JDBC経由で指定したSQLファイルを実行するスクリプトを利用し、自動化すること。

## 8. Docker Compose 設定

### 8.1 docker-compose.yml
MySQL と MinIO を Docker Compose で起動するための設定例：

```yaml
version: '3.8'

services:
  mysql:
    image: mysql:8
    container_name: mysql
    restart: always
    environment:
      MYSQL_ROOT_PASSWORD: rootpassword
      MYSQL_DATABASE: twitter_clone
      MYSQL_USER: user
      MYSQL_PASSWORD: userpassword
    ports:
      - "3306:3306"
    volumes:
      - mysql_data:/var/lib/mysql

  minio:
    image: minio/minio
    container_name: minio
    restart: always
    environment:
      MINIO_ROOT_USER: minioadmin
      MINIO_ROOT_PASSWORD: minioadmin
    command: server /data --console-address ":9001"
    ports:
      - "9000:9000"
      - "9001:9001"
    volumes:
      - minio_data:/data

volumes:
  mysql_data:
  minio_data:
```

## 9. DBマイグレーション

- **ツール**: Flyway を利用し、冪等性のある SQL でマイグレーションファイルを管理する。
- **格納場所**:  
  - マイグレーションファイルは `src/main/resources/db/migration` に配置する。
  - 必要に応じ、テスト環境用に `src/test/resources/db/migration` も利用可能とする。
- **運用方法**:  
  - Flyway はアプリケーション起動時に自動的にこのディレクトリからマイグレーションファイルを検出し、実行する。
  - 冪等性を担保した SQL スクリプトを用いて、スキーマの更新・追加を管理する。

## 10. DBマイグレーション

- **ツール**: Flyway を利用
- **格納場所**:  
  - マイグレーションファイルは `src/main/resources/db/migration` に配置する
  - 必要に応じ、テスト環境用は `src/test/resources/db/migration` に配置する
- **運用方法**:  
  - 冪等性のある SQL を用いてスキーマ更新やテーブル追加を記述する  
  - アプリケーション起動時、Flyway が自動的にこれらのマイグレーションファイルを検出して実行する
